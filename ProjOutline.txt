Project Overview
In this project, each team will identify a research question or business problem that they find genuinely interesting and use data to investigate it. The goal is not simply to analyze an existing dataset. The goal is to obtain data from real-world sources, prepare it for analysis, and communicate meaningful findings.

All core data work for the project must be completed in R. This includes data acquisition, cleaning, merging, transformation, and validation. The final visualizations and dashboard should be created in Tableau or Power BI.

Students should choose a topic they care about and build the dataset themselves. They should not rely on canned or pre-packaged datasets as the main basis for the project. Likely starting points include labor and jobs-related data, Inside Airbnb, and other public sources discussed in class.

The purpose of the project is to give students practice with the full data workflow. This includes defining a question, acquiring data, cleaning and integrating it, visualizing it in Tableau or Power BI, and communicating insights to both technical and non-technical audiences.

Data Requirements
Each team is expected to build a project dataset using at least three real-world data sources. Students should choose sources that are relevant to their research question or business problem and that require meaningful work to acquire, clean, and integrate.

Projects are expected to use at least two different acquisition methods. Common examples include:

web scraping
API access
structured extraction from unstructured data using an LLM
The most common project format will include one scraped source, one API-based source, and one source created through structured extraction from unstructured data. However, I am willing to work with teams whose research interests require a different combination of sources or methods.

The project should not rely primarily on canned, static, or pre-packaged datasets from sources such as Kaggle or GitHub. Students are expected to do the work of obtaining, cleaning, combining, and validating the data themselves.

Teams should use current or regularly updated data whenever possible. Good starting points may include data sources discussed in class, such as labor and jobs-related data, Inside Airbnb, and other public sources that support the chosen topic.

Technical Workflow Requirements
All core data work for the project must be completed in R. This includes data acquisition, cleaning, transformation, merging, and validation.

Each team must maintain a GitHub repository that contains the code used to build the final dataset. The repository should make it clear how the data was collected, processed, and combined. Code should be organized, readable, and reproducible.

At a minimum, the repository should include:

a README.md file that explains the project, data sources, and workflow
code or scripts for data acquisition
code or scripts for data cleaning and transformation
code or scripts for merging and preparing the final dataset
code or scripts for data validation and exploratory summaries
clear instructions for reproducing the pipeline
Your workflow should produce a final merged dataset that is tidy, technically correct, and consistent with the goals of the project. Teams are expected to document how they handled issues such as missing values, inconsistent formats, duplicate records, and mismatched keys.

Teams should also provide a data validation table for the merged dataset. This table should identify observations that violate expected rules, assumptions, or data quality checks. Examples include impossible values, duplicate entries, invalid categories, or failed joins.

In addition to building the final dataset, teams should use R to produce descriptive summaries of the data. This may include summary statistics, data quality checks, and exploratory visuals that help explain the structure and reliability of the dataset.

Important: Teams are responsible for following good data and code security practices. Use environment variables, configuration files excluded by.gitignore, or other secure methods to manage credentials. Do not publish API keys, access tokens, passwords, or other secrets in your GitHub repository. Teams will be penalized for exposing credentials or sensitive access information in a public repository.
The purpose of this requirement is to show that the team can build a complete and reproducible data pipeline, not just present final visuals.

Dashboard Requirements
Each team must create a final dashboard in Tableau or Power BI to communicate the main results of the project. The dashboard should be based on the final cleaned and merged dataset produced in R.

The dashboard should present the most important patterns, comparisons, and findings from the project in a clear and visually effective way. Teams should focus on helping the audience understand the big picture, the key relationships in the data, and the main insights that answer the research question or business problem.

The dashboard should be designed for communication, not just display. Visuals should be readable, relevant, and well organized. Labels, titles, filters, and annotations should help the audience interpret the results correctly.

If the dashboard is created in Tableau, teams are expected to use the Storyboard feature to organize and present the analysis. If the dashboard is created in Power BI, teams should organize the report in a similarly clear and logical way that supports storytelling and interpretation.

The first page of the dashboard, or the first story point if using Tableau Storyboard, should include the team member names and the team number.

The dashboard should include appropriate interactivity, such as filters, slicers, or drill-down features, when these features improve understanding. All visuals included in the dashboard should directly support the project question and final conclusions.

Teams must publish or share the final dashboard so that it can be accessed by the instructor. If using Tableau, the dashboard should be published through Tableau Public. If using Power BI, teams should share a working link to the dashboard.

The dashboard should support the final in-class presentation. It should help the team communicate business insights, major findings, and the broader significance of the project.

Recorded Technical Presentation
Each team must submit a recorded technical presentation using Loom or Zoom. The recording should be uploaded to YouTube as an unlisted video and shared with the instructor.

The recorded presentation should be 8 to 12 minutes long.

This presentation should focus on the technical side of the project. It should explain the question being studied, why the question is interesting, how the data was obtained, and how the final dataset was built.

At a minimum, the recorded presentation should address the following:

the research question or business problem
why the question is worth studying
the data sources used in the project
the acquisition methods used for each source
how the data was cleaned, transformed, merged, and validated in R
the structure of the final dataset
the novelty of the project and how it differs from simply analyzing an existing dataset
any important challenges, assumptions, or limitations in the data pipeline
The presentation should make clear that the team built the data pipeline themselves. It should show the logic of the workflow and explain how the final dataset supports the project goals. Note, I did not refer to any visuals here.

Teams do not need to turn this into a business presentation. The purpose of this recording is to document the technical quality of the project and the thinking behind the data pipeline.

In-Class Presentation
Each team will give an in-class presentation during the scheduled final exam period. The presentation should be 8 to 9 minutes long, followed by 1 to 2 minutes for questions. Every team member must participate in the presentation.

This presentation should focus on the big picture of the project rather than the technical details of the data pipeline. The goal is not to replicate the recorded technical presentation. Instead, the team should explain why the problem is important, what makes the question interesting or novel, what the visuals reveal, and what conclusions can be drawn from the analysis. Think of this as presenting your work to high-level executives.

At a minimum, the in-class presentation should address the following:

the research question or business problem
why the problem matters
what makes the question interesting, relevant, or novel
the key insights revealed by the dashboard and visuals
the main conclusions, implications, or takeaways from the project
If the dashboard is created in Tableau, teams are expected to use the Storyboard feature to support the presentation. If the dashboard is created in Power BI, teams should organize the report in a similarly clear and logical way that supports storytelling.

The purpose of this presentation is to communicate the value of the project to an audience. Teams should focus on clear interpretation, strong storytelling, and the significance of their findings.

Team Requirements
Students in ISA 401 are expected to work in teams on this project. The preferred team size is three students per team. Teams of up to four students are allowed. Students may choose their own team members.

Students in ISA 501 have similar project requirements, but they are expected to complete the project individually.

Teams should choose a project topic that reflects a shared interest and that can be completed successfully within the scope of the course. Because the project involves substantial work in data acquisition, cleaning, integration, validation, visualization, and presentation, teams should plan early and divide responsibilities thoughtfully.

All team members are expected to contribute meaningfully to the project. This includes participation in the data workflow, dashboard development, and preparation for both presentations.

Every team member must participate in the in-class presentation during the final exam period. Teams should make sure that each member plays an active role in presenting the project’s purpose, insights, and conclusions.

Teams are also responsible for coordinating their work professionally. This includes maintaining shared access to code, documenting progress, and making sure that all required materials are submitted in a complete and organized way.
